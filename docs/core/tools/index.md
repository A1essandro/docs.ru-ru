# <a name="net-core-commandline-interface-tools"></a>Средства интерфейса командной строки .NET Core

Интерфейс командной строки (CLI) .NET Core — это новая базовая кроссплатформенная цепочка инструментов для разработки приложений .NET Core. Она является базовой, так как это первый уровень, на основе которого могут строиться другие инструменты более высоких уровней, такие как интегрированные среды разработки (IDE), редакторы и оркестраторы сборки. 

Кроме того, она является межплатформенной по умолчанию и имеет одинаковую контактную зону на каждой поддерживаемой платформе. Это означает, что, если вы научились использовать средство, вы можете использовать его одинаковым образом с любой из поддерживаемых платформ. 

## <a name="installation"></a>Установка
Так же как и в случае с любыми другими средствами, этот пакет нужно установить на компьютере. В зависимости от сценария можно использовать либо собственные установщики CLI, либо скрипт оболочки для установки.

Собственные установщики предназначены в первую очередь для компьютеров разработчиков. Интерфейс CLI распространяется посредством собственных механизмов установки каждой поддерживаемой платформы, например пакетов DEB в Ubuntu или пакетов MSI в Windows. Эти установщики установят и настроят среду необходимым образом, чтобы пользователь мог сразу приступить к работе с CLI. Однако они требуют прав администратора на компьютере. Инструкции по установке см. на [странице, посвященной началу работы с .NET Core](https://aka.ms/dotnetcoregs).

Скрипты установки, в свою очередь, не требуют прав администратора. Однако они не устанавливают на компьютере необходимые компоненты; вам потребуется установить их вручную. Скрипты предназначены в первую очередь для настройки серверов сборки или установки средств без прав администратора (не забывайте об особенности, касающейся необходимых компонентов). Дополнительные сведения см. в [справочном разделе по скриптам установки](dotnet-install-script.md). Если вас интересует, как настроить интерфейс CLI на сервере сборки с непрерывной интеграцией (CI), обратитесь к разделу [Интерфейс CLI с серверами CI](using-ci-with-cli.md). 

По умолчанию интерфейс CLI устанавливается в параллельном режиме. Это означает, что на одном компьютере в любой момент времени могут сосуществовать разные версии средств CLI. То, как использовать правильную версию, более подробно описывается в разделе, посвященном [драйверу](#driver). 

### <a name="what-commands-come-in-the-box"></a>Какие команды входят в комплект поставки
По умолчанию устанавливаются следующие команды:

* [new](dotnet-new.md)
* [restore](dotnet-restore.md)
* [run](dotnet-run.md)
* [build](dotnet-build.md)
* [test](dotnet-test.md)
* [publish](dotnet-publish.md)
* [pack](dotnet-pack.md)

Кроме того, можно импортировать дополнительные команды для каждого отдельного проекта, а также добавить собственные команды. Более подробно это описывается в [разделе о расширяемости](#extensibility). 

## <a name="working-with-the-cli"></a>Работа с CLI

Перед тем как переходить к подробностям, давайте рассмотрим принципы работы с CLI в самом общем виде. В приведенном ниже примере используется несколько команд из стандартной установки CLI для инициализации нового простого консольного приложения, восстановления зависимостей, сборки приложения и последующего его запуска. 

```console
dotnet new
dotnet restore
dotnet build --output /stuff
dotnet /stuff/new.dll
```

Как видно в предыдущем примере, использование средств CLI подчиняется определенной схеме. В рамках этой схемы можно выделить три основных компонента каждой команды:

1. [драйвер (dotnet);](#driver)
2. [команда, или "глагол";](#the-verb)
3. [аргументы команды.](#the-arguments)

### <a name="driver"></a>Драйвер
Драйвер называется [dotnet](dotnet.md). Это первая часть вызываемой команды. Драйвер отвечает за две задачи:

1. запуск переносимых приложений;
2. выполнение команды.

Выполняемая задача зависит от того, что указано в командной строке. В первом случае указывается библиотека DLL переносимого приложения, которую должен выполнить драйвер `dotnet`, следующим образом: `dotnet /path/to/your.dll`. 

Во втором случае драйвер пытается вызвать указанную команду. Это приводит к запуску процесса выполнения команды CLI. Сначала драйвер определяет нужную версию средств. Версию можно задать в файле [global.json](global-json.md) с помощью свойства `version`sdkVersion. Если она недоступна, драйвер находит последнюю версию средств, установленную на диске, и использует ее. После определения версии он выполняет команду. 

### <a name="the-verb"></a>Команда
Команда служит для выполнения действия. `dotnet build` проводит сборку кода. `dotnet publish` публикует код. Команда реализуется как консольное приложение, имя которого соответствует следующему соглашению: `dotnet-{verb}`. Вся логика реализуется в консольном приложении, представляющем команду. 

### <a name="the-arguments"></a>Аргументы
Аргументы, указываемые в командной строке, передаются непосредственно в вызываемую команду. Например, если ввести `dotnet publish --output publishedapp`, аргумент `--output` будет передан в команду `publish`. 

## <a name="types-of-application-portability"></a>Типы переносимости приложений
CLI обеспечивает переносимость приложений двумя способами:

1. полностью переносимые приложения, которые могут выполняться везде, где установлена платформа .NET Core;
2. автономные развертывания.

Дополнительные сведения об обоих способах см. в разделе [Развертывание приложений .NET Core](../deploying/index.md). 

## <a name="migration-from-dnx"></a>Переход с DNX
Если вы использовали среду DNX в версии-кандидате 1 .NET Core 1.0, у вас может возникнуть вопрос, что произошло с ней и как новые средства соотносятся со средствами DNX. Если говорить кратко, средства DNX были заменены средствами CLI .NET Core. Если у вас имеются существующие проекты или вы просто хотите узнать, как сопоставляются команды, то подробные сведения можно получить в разделе [Переход с DNX на CLI](../migrating-from-dnx.md). 

## <a name="extensibility"></a>Расширение среды
Конечно, не каждое средство, которое вы могли использовать в своем рабочем процессе, входит в базовый набор средств CLI. Однако интерфейс CLI .NET Core предлагает модель расширяемости, которая позволяет указывать дополнительные средства для проектов. Дополнительные сведения см. в разделе [Модель расширяемости CLI .NET Core](extensibility.md).
