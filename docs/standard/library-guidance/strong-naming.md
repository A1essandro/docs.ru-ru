# <a name="strong-naming"></a>Строгое именование

Строгое именование ссылается на подпись сборки с ключом, создания [сборки со строгими именами](../../framework/app-domains/strong-named-assemblies.md). При сборки со строгим именем, он создает уникальное удостоверение, основанное на номер версии имя и сборку, и поможет избежать конфликтов сборок.

Строгое именование недостаток в том, что .NET Framework на Windows позволяет strict загрузки сборок, после сборки есть строгое имя. Ссылку на сборку со строгим именем должны точно совпадать с версией, ссылается на сборку, чтобы заставлять разработчиков придумывать [Настройка перенаправления привязки](../../framework/configure-apps/redirect-assembly-versions.md) при использовании сборки:

```xml
<configuration>
   <runtime>
      <assemblyBinding xmlns="urn:schemas-microsoft-com:asm.v1">
         <dependentAssembly>
            <assemblyIdentity name="myAssembly" publicKeyToken="32ab4ba45e0a69a1" culture="neutral" />
            <bindingRedirect oldVersion="1.0.0.0" newVersion="2.0.0.0"/>
         </dependentAssembly>
      </assemblyBinding>
   </runtime>
</configuration>
```

Разработчикам .NET жалуются на строгое именование, что они обычно жалуется при строгом загрузки сборки. К счастью эту проблему изолирован на .NET Framework. .NET core, Xamarin, универсальной платформы Windows и большинство других реализаций .NET нет строгой загрузки сборки и удаляет основной недостаток использования строгих имен.

Одним важным аспектом строгих имен — это популярный: надежный с именем сборки могут только одну ссылку других надежный с именем сборки. Если библиотека не надежный с именем, затем был исключен разработчиков, создающих приложения или библиотеки, которому требуется строгое именование от его использования.

Приведены преимущества строгое именование.

1. Сборки можно ссылаться и использовать в других сборках со строгими именами.
2. Сборки могут храниться в глобальный кэш сборок (GAC).
3. Сборка может быть загружена параллельно с другими версиями сборки. Загрузка сборки Side-by-side часто требуется приложений с помощью подключаемого модуля архитектур.

## <a name="create-strong-named-net-libraries"></a>Создайте надежный с именем библиотеки .NET

Следует строгое именование .NET библиотек открытым исходным кодом. Строгое именование сборки гарантирует, большинство пользователей можно использовать его и строгом загрузки только сборок влияет на .NET Framework.

> [!NOTE]
> Это руководство относится к свободно распространяемые библиотеки .NET, таких как библиотеки .NET опубликованы на сайте NuGet.org. Строгое именование не является обязательным для большинства приложений .NET и не должна осуществляться по умолчанию.

**Рассмотрите ВОЗМОЖНОСТЬ ✔️** строгого именования сборки библиотеки.

**Рассмотрите ВОЗМОЖНОСТЬ ✔️** проверки в ключ, используемый для строгого имени в систему управления версиями.

> Общедоступные ключ позволяет разработчикам измените и перекомпилируйте исходный код библиотеки с тем же ключом.

> [!IMPORTANT]
> При необходимости шифрования удостоверений [Authenticode](/windows-hardware/drivers/install/authenticode) и [Подписание пакета NuGet](/nuget/create-packages/sign-a-package) рекомендуются. Строгое именование не следует по соображениям безопасности.

**Рассмотрите ВОЗМОЖНОСТЬ ✔️** увеличение версии сборки на изменения только основной номер версии, помогающие пользователям сократить переадресации привязок, и как часто они обновлены.

> Дополнительные сведения о [управление версиями и версию сборки](./versioning.md#assembly-version).

**❌ НЕ** добавления, удаления или изменения ключа строгого именования.

> Изменение сборки строгого именования ключ изменяет удостоверение сборки и разбивает скомпилированный код, который его использует. Дополнительные сведения см. в разделе [двоичных критические изменения](./breaking-changes.md#binary-breaking-change).

**❌ НЕ** публикации со строгим именем и не-со строгими именами версий библиотеки. Например, `Contoso.Api` и `Contoso.Api.StrongNamed`.

> Публикация две вилки пакеты разработчика в экономичном системы. Кроме того в зависимости от того, оба пакета приложения заканчивается разработчика могут возникнуть конфликты имя типа. Что касается .NET это разные типы в разных сборках.
