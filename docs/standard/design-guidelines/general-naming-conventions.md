# Общие соглашения об именовании
В этом разделе описаны общие соглашения об именовании, связанные с выбором word, инструкции по использованию сокращения и акронимы и рекомендации о том, как избежать использования имен конкретного языка.  
  
## Выбор слов  
 **✓ сделать** выберите легко читаемые имена идентификаторов.  
  
 Например, свойство с именем `HorizontalAlignment` является английский понятным, чем `AlignmentHorizontal`.  
  
 **✓ сделать** предпочтение краткости удобочитаемости.  
  
 Имя свойства `CanScrollHorizontally` лучше, чем `ScrollableX` \(непрозрачными ссылка на ось x\).  
  
 **X не** использовать знаки подчеркивания, дефисы и другие символы.  
  
 **X не** использовать венгерскую нотацию.  
  
 **ИЗБЕЖАТЬ X** использования идентификаторов, конфликтующих с ключевыми словами широко используемые языки программирования.  
  
 Согласно правило 4 из общей спецификации \(CLS\) все языки, совместимые необходимо предоставить механизм, обеспечивающий доступ к именованных элементов, используйте ключевое слово этого языка в качестве идентификатора. C\#, например, используется знак как механизм escape в данном случае @. Тем не менее он по\-прежнему хороший способ избежать распространенные ключевые слова, так как это гораздо сложнее использовать метод с escape\-последовательности, чем без него.  
  
## С помощью сокращения и акронимы  
 **X не** использовать аббревиатуры или сокращения как часть имени идентификатора.  
  
 Например, используйте `GetWindow` вместо `GetWin`.  
  
 **X не** использовать акронимы, которые не являются общепринятым и даже в том случае, если они находятся, только при необходимости.  
  
## Чтобы избежать имен, зависящих от языка  
 **✓ сделать** использование семантически значимых имен вместо зарезервированные слова языка для имен типов.  
  
 Например `GetLength` является именем лучше, чем `GetInt`.  
  
 **✓ сделать** используйте имя универсального типа CLR, а не имя конкретного языка, в редких случаях, когда идентификатор не имеет семантического значения за пределами своего типа.  
  
 Например, преобразование в метод <xref:System.Int64> должен быть назван `ToInt64`, а не `ToLong` \(поскольку <xref:System.Int64> — это имя среды CLR для C\#\-псевдоним `long`\). В следующей таблице представлены некоторые базовые типы данных с помощью имен типов среды CLR \(а также соответствующие имена типов для C\#, Visual Basic и C\+\+\).  
  
|C\#|Visual Basic|C\+\+|CLR|  
|---------|------------------|-----------|---------|  
|**sbyte**|**SByte**|**char**|**SByte**|  
|**byte**|**Byte**|**unsigned char**|**Byte**|  
|**short**|**Краткое**|**short**|**Int16**|  
|**ushort**|**UInt16**|**unsigned short**|**UInt16**|  
|**int**|**Целое число**|**int**|**Int32**|  
|**uint**|**UInt32**|**unsigned int**|**UInt32**|  
|**long**|**Long**|**\_\_int64**|**Int64**|  
|**ulong**|**UInt64**|**unsigned \_\_int64**|**UInt64**|  
|**float**|**Single**|**float**|**Single**|  
|**double**|**Double**|**double**|**Double**|  
|**bool**|**Boolean**|**bool**|**Boolean**|  
|**char**|**Char**|**wchar\_t**|**Char**|  
|**string**|**Строка**|**Строка**|**Строка**|  
|**object**|**Объект**|**Объект**|**Объект**|  
  
 **✓ сделать**  использование обычных имен, таких как `value` или `item`, вместо того чтобы повторение имени типа в редких случаях, когда идентификатор не имеет семантического значения и тип параметра не имеет значения.  
  
## Именование новых версий существующих API  
 **✓ сделать** использовать именем, похожим на старый API при создании новой версии существующего API.  
  
 Это позволяет выделить связь между API\-интерфейсы.  
  
 **✓ сделать** необходимости добавления суффикса вместо префикса для указания новой версии существующего API.  
  
 Это упростит обнаружения при просмотре документации, или с помощью Intellisense. Старые версии API\-интерфейса будут организованы рядом новых интерфейсов API, поскольку большинство браузеров и Intellisense Показывать идентификаторы в алфавитном порядке.  
  
 **✓ Рассмотрите ВОЗМОЖНОСТЬ** с помощью идентификатора совершенно новый, но может применяться, вместо добавления суффикса и префикса.  
  
 **✓ сделать** использовать числовой суффикс для указания новой версии существующего API, особенно в том случае, если только имя, которое имеет смысл \(т. е. Если это отраслевой стандарт\) существующее имя API, и при добавлении все значимые суффикс \(или изменение имени\) не соответствующий параметр.  
  
 **X не** «Ex» \(или аналогичной\) суффикс идентификатора позволяет отличить его от более ранней версии, такой же API.  
  
 **✓ сделать** используйте суффикс «64» при выходе версии API, которые работают в 64\-разрядное целое \(длинное целое\), а не 32\-разрядное целое число. Необходимо выбрать такой подход при наличии существующих 32\-разрядных API; не делайте этого для нового API с 64\-разрядной версии.
