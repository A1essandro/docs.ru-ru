# <a name="choosing-between-net-core-and-net-framework-for-server-apps"></a>Выбор между .NET Core и .NET Framework для серверных приложений

Серверные приложения на платформе .NET можно разрабатывать в двух поддерживаемых средах выполнения: .NET Framework и .NET Core. В них используется множество одинаковых компонентов этой платформы, а код можно использовать как в одной среде, так и в другой. Однако между этими двумя средами также существуют и фундаментальные различия. Поэтому ваш выбор будет зависеть от поставленной задачи.  В этой статье содержатся рекомендации по использованию каждой из двух сред выполнения.

Используйте среду .NET Core для создания серверных приложений в следующих случаях:

* для создания кроссплатформенных решений;
* для создания решений, ориентированных на микрослужбы;
* при использовании контейнеров Docker;
* если нужны масштабируемые системы с высокой производительностью;
* для создания приложений с поддержкой разных версий .NET.

Используйте среду .NET Framework для создания серверных приложений в следующих случаях:

* приложение в настоящий момент использует среду .NET Framework (мы рекомендуем расширять приложения, а не переносить их в другую среду);
* требуются сторонние библиотеки .NET или пакеты NuGet, недоступные для .NET Core;
* требуются технологии .NET, недоступные для .NET Core;
* требуется платформа, не поддерживающая .NET Core.

## <a name="when-to-choose-net-core"></a>Случаи использования .NET Core

Ниже более детально описаны указанные ранее причины для выбора платформы .NET Core.

### <a name="cross-platform-needs"></a>Создание кроссплатформенных приложений

Если вам нужно создать приложение (веб-приложение или службу) с поддержкой разных платформ (Windows, Linux и macOS), мы рекомендуем использовать .NET Core.

В среде .NET Core также можно использовать упомянутые ранее операционные системы в качестве рабочих станций для разработки. Visual Studio предоставляет интегрированную среду разработки (IDE) для Windows.  Кроме того, редактор Visual Studio Code можно использовать в macOS, Linux и Windows. Он полностью поддерживает .NET Core, в том числе IntelliSense и отладку. Для разработки на платформе .NET Core также можно использовать большинство сторонних редакторов, например Sublime, Emacs, VI. Поддержку редактора IntelliSense можно добавить с помощью проекта [Omnisharp](http://www.omnisharp.net/) с открытым исходным кодом. Платформа .NET Core полагается не только на редакторы. Приложения здесь можно создавать непосредственно с помощью средств командной строки, доступных для всех поддерживаемых платформ.

### <a name="microservices-architecture"></a>Архитектура микрослужб

Если вам нужна ориентированная на микрослужбы система, состоящая из нескольких независимых масштабируемых с течением времени микрослужб с отслеживанием или без отслеживания состояния, тогда .NET Core — это лучший вариант. Платформа .NET Core имеет небольшой размер, а рабочую область API можно уменьшить до объема микрослужбы. Архитектура микрослужб также позволяет сочетать технологии разных служб, что, в свою очередь, предоставляет возможность постепенно добавлять новые микрослужбы в .NET Core. Эти службы используются параллельно с другими службами и микрослужбами, созданными на основе .NET Framework, Java, Ruby или других монолитных технологий.

Пользователям на выбор предоставляется множество инфраструктурных платформ. Для больших и сложных систем микрослужб можно использовать [Azure Service Fabric](https://azure.microsoft.com/services/service-fabric/), а для микрослужб без отслеживания состояния — другие продукты, например [Azure App Service](https://azure.microsoft.com/services/app-service/). Микрослужбы на основе Docker поддерживают любые подходы, как описано дальше. Все эти платформы поддерживают .NET Core и идеально подходят для размещения микрослужб.

### <a name="containers"></a>Контейнеры

Контейнеры обычно используются вместе с архитектурой микрослужб, хотя их также можно использовать, чтобы поместить в контейнер веб-приложения или службы на базе любого архитектурного шаблона. Среду .NET Framework можно использовать для контейнеров Windows, но упрощенные возможности и модульный принцип среды .NET Core делают ее оптимальной для контейнеров. При создании и развертывании контейнера размер его образа гораздо меньше в среде .NET Core. Так как эта среда поддерживает разные платформы, серверные приложения можно развертывать, например, в контейнеры Docker в Linux.

Затем эти контейнеры можно размещать в собственной инфраструктуре Linux или Windows. Кроме того, с помощью облачной службы, например [Службы контейнеров Azure](https://azure.microsoft.com/services/container-service/), можно выполнять оркестрацию и масштабировать приложения на основе контейнеров, а также управлять ими в облаке.

### <a name="a-need-for-high-performance-and-scalable-systems"></a>Масштабируемые системы с высокой производительностью

Если для вашей системы требуется максимальная производительность и возможности масштабирования, мы рекомендуем использовать среды .NET Core и ASP.NET Core. Среда ASP.NET Core превышает производительность ASP.NET в 10 раз и поддерживает другие популярные отраслевые технологии микрослужб, такие как сервлеты Java, Go и Node.js.

Это особенно важно для архитектур микрослужб, где могут выполняться сотни микрослужб. Среда ASP.NET Core позволяет уменьшить количество серверов и виртуальных машин, необходимых для системы, и в конечном счете сократить расходы на архитектуру и размещение.

### <a name="a-need-for-side-by-side-of-net-versions-per-application-level"></a>Создание приложений с поддержкой разных версий .NET

Если требуется установить приложения с зависимостями в разных версиях платформ .NET, используйте среду .NET Core, обеспечивающую полный параллелизм. За счет простой параллельной установки разных версий .NET Core на одной виртуальной машине пользователи могут размещать на том же самом сервере несколько служб, разработанных в разных версиях .NET Core. Это позволяет устранить риски и сократить расходы на обновление приложений и ИТ-операции.

## <a name="when-to-choose-net-framework"></a>Случаи использования .NET Framework

Хотя среда .NET Core предоставляет значительные преимущества для новых приложений и шаблонов приложений, мы рекомендуем использовать среду .NET Framework для большинства имеющихся сценариев и, исходя из этого, — для всех серверных приложений.

### <a name="current-net-framework-applications"></a>Готовые приложения .NET Framework

В большинстве случаев вам не потребуется переносить готовые приложения в среду .NET Core. Взамен мы советуем использовать .NET Core для расширения готовых приложений, например для написания новой веб-службы в ASP.NET Core.

### <a name="a-need-to-use-third-party-net-libraries-or-nuget-packages-not-available-for-net-core"></a>Поддержка сторонних библиотек .NET или пакетов NuGet, недоступных для .NET Core

Библиотеки .NET Standard позволяют совместно использовать код в различных средах выполнения .NET, в том числе в .NET Core. Благодаря тому, что библиотеки .NET Standard 2.0 значительно расширяют API-интерфейс .NET Core, а приложения .NET Core могут непосредственно использовать имеющиеся библиотеки .NET Framework, процесс совместного использования стал еще проще. Этот переход занимает некоторое время, поэтому перед принятием решения мы рекомендуем пересмотреть библиотеки, необходимые для приложения.

### <a name="a-need-to-use-net-technologies-not-available-for-net-core"></a>Поддержка технологий .NET, недоступных для .NET Core

Некоторые технологии .NET Framework недоступны в среде .NET Core. Поддержка некоторых из них будет добавлена в последующих выпусках .NET Core, однако остальные технологии не применяются к новым шаблонам приложений, используемым в среде .NET Core, и будут недоступными всегда. Ниже приведен список самых распространенных технологий, которые недоступны в .NET Core 1.0.

* Приложения веб-форм ASP.NET. Веб-формы ASP.NET доступны только в среде .NET Framework, поэтому этот сценарий нельзя реализовать в среде ASP.NET Core или .NET Core. В настоящее время добавление веб-форм ASP.NET в среду .NET Core не планируется.

* Приложения веб-страниц ASP.NET. Веб-страницы ASP.NET не добавлены в среду ASP.NET Core 1.0. Однако их планируют добавить в следующем выпуске. Дополнительные сведения см. в [плане развития .NET Core](https://github.com/aspnet/Home/wiki/Roadmap).

* Реализация сервера и клиента ASP.NET SignalR. В выпуске .NET Core 1.0 от июня 2016 года ASP.NET SignalR недоступен (ни клиент, ни сервер). Однако их планируют добавить в следующем выпуске. Дополнительные сведения см. в [плане развития .NET Core](https://github.com/aspnet/Home/wiki/Roadmap). Предварительная пробная версия библиотеки для [сервера](https://github.com/aspnet/SignalR-Server) и [клиента](https://github.com/aspnet/SignalR-Client-Net) доступна в репозиториях GitHub.

* Реализация служб WCF. Хотя в среду .NET Core и добавлена [библиотека клиента WCF](https://github.com/dotnet/wcf), позволяющая использовать службы WCF, по состоянию на июнь 2016 г. сервер WCF можно реализовать только в среде .NET Framework. Этот сценарий не входит в текущий план для .NET Core, но он будет рассмотрен в будущем.

* Службы, связанные с рабочим процессом. Службы Windows Workflow Foundation (WF), Workflow Services (WCF и WF в одной службе) и WCF Data Services (ранее службы данных ADO.NET) доступны только в среде .NET Framework, и их добавление в .NET Core не планируется.

* Поддержка языков. В настоящее время в среде .NET Core отсутствуют средства для поддержки языков Visual Basic и F#, но их поддержка будет добавлена в Visual Studio 2017 и более поздние версии.

Кроме платформ, указанных в официальной стратегии, существуют и другие платформы, которые можно перенести в среду .NET Core. Полный список см. в репозитории CoreFX. Эти платформы помечены как [port-to-core](https://github.com/dotnet/corefx/issues?q=is%3Aopen+is%3Aissue+label%3Aport-to-core). Обратите внимание, что корпорация Майкрософт не намеревается добавлять эти компоненты в среду .NET Core. Компоненты, добавленные в список, отражают желания пользователей сообщества. Тем не менее, если вас заинтересовал любой из перечисленных выше компонентов, примите участие в обсуждениях на сайте GitHub и выразите свое мнение. Если вы думаете, что чего-то не хватает, [отправьте новый вопрос в репозиторий CoreFX](https://github.com/dotnet/corefx/issues/new).

### <a name="a-need-to-use-a-platform-that-doesnt-support-net-core"></a>Использование платформы, не поддерживающей .NET Core

Некоторые платформы Майкрософт и платформы сторонних поставщиков не поддерживают среду .NET Core. Например, для некоторых служб Azure, таких как Service Fabric Reliable Services с отслеживанием состояния и Service Fabric Reliable Actors, требуется среда .NET Framework. Некоторые другие службы предоставляют пакеты SDK, недоступные в среде .NET Core. Это временные обстоятельства, так как все службы Azure используют среду .NET Core. В то же время в качестве альтернативы клиентскому пакету SDK всегда можно использовать REST API.
