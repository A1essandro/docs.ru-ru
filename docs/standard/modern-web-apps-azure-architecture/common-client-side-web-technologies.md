## <a name="summary"></a>Сводка

Приложения ASP.NET Core, веб-приложений, и они обычно полагаются на стороне клиента веб-технологий, как HTML, CSS и JavaScript. Разделив содержимое страницы (HTML) в макете и стилей (CSS) и его поведение (с помощью JavaScript), сложных веб-приложений можно использовать принцип значимость. Будущие изменения в структуру, конструктора или поведение приложения могут осуществляться более легко когда эти проблемы не тесно связаны.

Хотя HTML и CSS, относительно стабильны, JavaScript, с помощью платформы приложений и разработчикам программ могут работать с для создания веб-приложений, развивается крайне сжатыми: всего скоростью. В этой главе рассматривается несколько способов JavaScript используется веб-разработчиками в процессе разработки приложений, как общий обзор угловой и реагирование на них библиотеках на стороне клиента.

## <a name="html"></a>HTML

Код HTML (HyperText Markup Language) является стандартный язык разметки используется для создания веб-страниц и веб-приложений. Его элементы формы являются составными частями страниц, представляющий форматированный текст, изображения, входных и других структур. Когда браузер выполняет запрос на URL-адрес ли выборка приложения или страницы, первое, что именно возвращаемый является HTML-документа. Этот документ HTML может ссылаться или включить дополнительные сведения о его внешнего вида и макета в виде CSS или поведения в виде объекта JavaScript.

## <a name="css"></a>CSS

CSS (каскадные таблицы стилей) используется для управления, внешний вид и расположение элементов HTML. Стили CSS можно применить непосредственно к HTML-элемент, определяются по отдельности на одной странице или определен в отдельном файле и ссылается страница. Cascade стили, в зависимости от того, как они используются для выбора данного элемента HTML. Для экземпляра стиля могут применяться ко всему документу, но может быть переопределено стиль, который применяется к определенному элементу. Аналогично стиль конкретного элемента был бы переопределен стилем, применяется класс CSS, который был применен к элементу, который в свою очередь, был бы переопределен в стиле, предназначенные для конкретного экземпляра этого элемента (через его идентификатор). ### <a name="css-preprocessors"></a>CSS-Препроцессоры

Таблицы стилей CSS не поддерживают условную логику, переменные и другие возможности языка программирования. Таким образом большие таблицы стилей часто содержат часто, как тот же цвет, шрифт или другого параметра применяется к много разновидностей элементы HTML и CSS-классы. CSS-препроцессоры может помочь вашей таблицы стилей выполните [ТОНЕРА принцип](http://deviq.com/don-t-repeat-yourself/) , добавляя поддержку переменных и логики.

Самые популярные CSS-препроцессоры являются Sass и меньше. Оба расширения CSS и обратно совместимы с ним, это означает, что обычный CSS-файл является допустимым Sass или файл. Sass выполняется на основе Ruby и меньше — на основе JavaScript и обычно работают как часть процесса локальной разработки. Оба имеют командной строки средств поддержки доступны, а также встроенных в Visual Studio для выполнения их с помощью Gulp или Grunt задач.

## <a name="javascript"></a>JavaScript

JavaScript — динамические, интерпретируемый язык программирования, который был стандартизирован в спецификации языка ECMAScript. Это язык программирования, веб-узла. Как и CSS JavaScript можно определить как атрибуты в HTML-элементов, как блоки скрипта на одной странице или в отдельных файлах. Так же, как CSS обычно рекомендуется организовать JavaScript в отдельных файлах предотвращает их запятыми максимально HTML, найденных на отдельные веб-страницы или представления приложения.

При работе с кодом JavaScript в веб-приложения, существует несколько задач, которые часто необходимо будет выполнить.

-   При выборе элемента HTML и извлечение или обновление его значения

-   Запросы веб-API для данных

-   Отправив команду на веб-API (и отвечать на обратный вызов, результат ее выполнения)

-   Выполнение проверки

Можно выполнить все эти задачи с помощью JavaScript сама по себе, но существует множество библиотек для облегчения этих задач. Одним из первой и наиболее успешных эти библиотеки является jQuery, который продолжает быть популярной упрощению этих задач на веб-страницах. Для одной страницы приложений (SPAs) jQuery не предоставляет многие из необходимых компонентов, предлагающий угловой и реагирование на них.

### <a name="legacy-web-apps-with-jquery"></a>Устаревшие веб-приложений с помощью jQuery

Несмотря на то что Древние стандартами framework JavaScript, jQuery продолжает быть очень часто используемые библиотеки для работы с HTML и CSS и создания приложений, которые выполняют вызовы AJAX для веб-API. Однако jQuery функционирует на уровне браузера объектной модели документа (DOM) и по умолчанию предлагает только императивного вместо декларативных, модель.

Например предположим, что если значение текстового поля превышает 10, элемент на странице следует сделать видимой. В jQuery это обычно реализуется путем написания обработчика событий с кодом, который будет проверять значение в текстовое поле и настроить видимость для целевого элемента, на основе этого значения. Этот подход императивный, основанная на коде. Другой структуре вместо может использовать привязку данных для привязки видимости элемента значение текстового поля декларативно. Не требуется написания кода, но вместо этого требуется только декорирования элементы, используемые с атрибутами привязки данных. По мере более сложные поведения клиента привязки данных приближается к часто приводят к более простые решения меньше кода и сложностью условных.

### <a name="jquery-vs-a-spa-framework"></a>jQuery vs SPA Framework

| **Коэффициент** | **jQuery** | **Угловое**|
|--------------------------|------------|-------------|
| Создает уровень абстракции для модели DOM | **Да** | **Да** |
| Поддержка AJAX | **Да** | **Да** |
| Декларативной привязки данных | **No** | **Да** |
| Маршрутизации в MVC-стиль | **No** | **Да** |
| Использование шаблонов | **No** | **Да** |
| Прямая ссылка маршрутизации | **No** | **Да** |

Большинство функций, которые jQuery само по себе не имеет могут добавляться с добавлением другие библиотеки. Тем не менее SPA платформы наподобие угловая поддерживает эти возможности интеграции образом, так как она предназначена с ними учитывать с самого начала. Кроме того jQuery — это очень императивного библиотека, это означает, что необходимости вызывать функции, jQuery, чтобы выполнять никаких действий с jQuery. Большая часть работы и функции, которые SPA платформы, обеспечивают можно сделать декларативно, требующие без фактического написания кода.

Привязка данных является отличным примером. В jQuery обычно это только занимает одну строку кода для получения значения элемента DOM, или задать значение элемента. Тем не менее необходимо написать следующий код в любое время, необходимо изменить значение элемента, а иногда это происходит в несколько функций на странице. Другой распространенный пример — видимость элемента. JQuery может возникнуть множества разных мест, где можно написать код для управления ли определенные элементы были видны. В каждый из этих случаев, когда использование привязки данных код не требуется записать. Нужно просто привязать значение или видимость для элементов *viewmodel* на странице и изменения, viewmodel автоматически отражались в связанных элементах.

### <a name="angular-spas"></a>Угловое SPAs

AngularJS быстро стал одним из наиболее популярных платформ JavaScript в мире. С углового 2, команда перестраивается framework с нуля вверх (с помощью [TypeScript](https://www.typescriptlang.org/)) и бренда из AngularJS, чтобы просто угловое. В настоящее время версии 4, угловая продолжает быть надежную платформу для построения приложений на одной странице.

Угловое-приложений из компонентов. Компоненты объединять шаблоны HTML с специальные объекты и управлять часть страницы. Простого компонента из документов угловая показан ниже:

```js
import { Component } from '@angular/core';

@Component({
    selector: 'my-app',
    template: `<h1>Hello {{name}}</h1>`
})

export class AppComponent { name = 'Angular'; }
```

Компоненты определяются с помощью @Component декоратора функции, которая принимает метаданные о компоненте. Свойство выбора указывает идентификатор элемента на странице, где отображается этот компонент. Свойства шаблона представляет собой простой шаблон HTML, который включает заполнитель, который соответствует свойству имя компонента, определенные в последней строке.

При работе с компонентами и шаблонов, а не элементами DOM, угловое приложения могут работать на более высоком уровне абстракции и меньше общего кода, чем приложения, написанные с использованием только JavaScript (также называется «ванильные JS») или с jQuery. Угловая также упорядочивает некоторые на способы организации файлы клиентских скриптов. По соглашению углового приложения используют структуру общих папок с файлами скриптов модуль и компонент находится в папке приложения. Угловое скриптов с построение, развертывание и тестирование приложения обычно расположены в папке более высокого уровня.

Угловая также обеспечивает значительные использование средства командной строки (CLI). Начало угловую разработки локально (при условии, что уже установлен npm и git) состоит из просто клонированием репозитория в GitHub и работает \`установки npm\` и \`npm начала\`. Кроме того угловая поставляется свой собственный интерфейс командной строки средства, которое можно создавать проекты, добавление файлов и помочь с задачами тестирования, объединение и развертывания. Этот интерфейс командной строки средства удобство использования обеспечивает угловая особенно совместимость с ASP.NET Core, который также включает отличную поддержку CLI.

Корпорация Майкрософт разработала приложении ссылку [eShopOnContainers](http://aka.ms/MicroservicesArchitecture), который включает в себя реализацию углового SPA. Это приложение включает угловых модулей для управления Интернет-магазина покупательская корзина, загрузка и отображение элементов в каталоге и обработка создания заказа. Можно просмотреть и загрузить пример приложения из [GitHub](https://github.com/dotnet-architecture/eShopOnContainers/tree/master/src/Web/WebSPA).

### <a name="react"></a>реагирование на них

В отличие от момента, который предоставляет полной реализации шаблона Model-View-Controller, реагирование на них связано только с представлениями. Он не является структурой, просто библиотека, поэтому для построения SPA вам потребуется использовать дополнительные библиотеки.

Один из самых важных функций реагирование на них является использование виртуального DOM. Виртуальный DOM предоставляет несколько преимуществ, включая производительности (виртуальный DOM можно оптимизировать, должны быть обновлены какие части фактического DOM) и пригодности для тестирования (не требуется браузер для тестирования реагирование на них и взаимодействия с виртуальной DOM) реагирование на них.

Реагирование на них также очень редко как она работает с HTML. Вместо того чтобы использовать строгое разделение между кодом и разметки (со ссылками на JavaScript, возможно появление в атрибутах HTML), реагируют добавляет HTML непосредственно в коде JavaScript в качестве JSX. JSX — HTML синтаксис, можно скомпилировать до чисто JavaScript. Пример:

```js
<ul>
{ authors.map(author =>
    <li key={author.id}>{author.name}</li>
)}
</ul>
```

Если вы уже знаете JavaScript, обучения реагирование на них должен быть простым делом. Нет немногим обучения или специальный синтаксис, связанных как с момента или других популярных библиотек.

Из-за реагирование на них не полной framework, обычно нужно других библиотек для обработки таких вещей, как маршрутизации, веб-вызовы API и управление зависимостями. Является хорошо, можно выбрать лучший библиотеки для каждого из них, но недостаток заключается в том, что необходимо сделать все эти решения и убедитесь, что все библиотеки выбранного работают вместе, по окончании. Если вы хотите хорошей отправной точкой, можно использовать начального набора, как реагировать Slingshot, prepackages набор совместимых библиотек, а также реагирование на них.

### <a name="choosing-a-spa-framework"></a>Выбор SPA Framework

При рассмотрении какие платформы JavaScript будет работать наилучшим образом для поддержки вашего SPA, имейте в виду следующее:

-   Команда является знакомы платформы и его зависимости (включая TypeScript в некоторых случаях)?

-   Как opinionated — это платформа, и вы согласны с его способе работы по умолчанию?

-   Он (или библиотека дополнительное) включает все возможности, необходимые для приложения?

-   Это хорошо документированы?

-   Как активный сообществом? Создание новых проектов построены с ним?

-   Как активный его основные команды? Проблемы, разрешенных и новые версии поставляются регулярно?

Платформы JavaScript по-прежнему развивать крайне сжатыми: всего скорость. Используйте рекомендации, приведенные выше, чтобы минимизировать риск выборе платформы, который позднее будет отменить, зависимость. Если вы особенно риск к снижению возможных, рассмотрите возможность платформа, которая обеспечивает поддержку коммерческих и/или разрабатывается крупного предприятия.
