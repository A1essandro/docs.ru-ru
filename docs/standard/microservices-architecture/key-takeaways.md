<span data-ttu-id="2d898-118">**Проектирование на основе домена и SOA.**</span><span class="sxs-lookup"><span data-stu-id="2d898-118">**Domain-driven design and SOA.**</span></span> <span data-ttu-id="2d898-119">Шаблоны архитектуры микрослужбами являются производными от сервисноориентированной архитектуре (SOA) и разработки на основе домена (DDD).</span><span class="sxs-lookup"><span data-stu-id="2d898-119">The microservices architecture patterns derive from service-oriented architecture (SOA) and domain-driven design (DDD).</span></span> <span data-ttu-id="2d898-120">При проектировании и разработке микрослужбами для сред с развивающейся бизнес-правила формирования определенного домена, очень важно учитывать подходы DDD учетной записи и шаблоны.</span><span class="sxs-lookup"><span data-stu-id="2d898-120">When you design and develop microservices for environments with evolving business rules shaping a particular domain, it is important to take into account DDD approaches and patterns.</span></span>

<span data-ttu-id="2d898-121">**Микрослужбами трудности.**</span><span class="sxs-lookup"><span data-stu-id="2d898-121">**Microservices challenges.**</span></span> <span data-ttu-id="2d898-122">Микрослужбами предоставляют мощные возможности, как независимые развертывания, границы строгого подсистемы и разнообразие технологии.</span><span class="sxs-lookup"><span data-stu-id="2d898-122">Microservices offer many powerful capabilities, like independent deployment, strong subsystem boundaries, and technology diversity.</span></span> <span data-ttu-id="2d898-123">Тем не менее они также создают новые проблемы, связанные с разработки распределенных приложений, таких как фрагментации и независимые данные модели, гибкой связи между микрослужбами, итоговая согласованность и сложности, полученный в результате Агрегация сведений в журнал и мониторинга из нескольких микрослужбами.</span><span class="sxs-lookup"><span data-stu-id="2d898-123">However, they also raise many new challenges related to distributed application development, such as fragmented and independent data models, resilient communication between microservices, eventual consistency, and operational complexity that results from aggregating logging and monitoring information from multiple microservices.</span></span> <span data-ttu-id="2d898-124">Эти аспекты ввести более высокий уровень сложности, чем традиционные монолитные приложения.</span><span class="sxs-lookup"><span data-stu-id="2d898-124">These aspects introduce a higher level of complexity than a traditional monolithic application.</span></span> <span data-ttu-id="2d898-125">В результате только конкретных сценариев, подходят для приложений на основе микрослужбы.</span><span class="sxs-lookup"><span data-stu-id="2d898-125">As a result, only specific scenarios are suitable for microservice-based applications.</span></span> <span data-ttu-id="2d898-126">К ним относятся больших и сложных приложений с несколькими подсистемами развивающегося; в таких случаях стоит инвестиции в более сложные архитектуры программного обеспечения, так как это обеспечивает лучшие долгосрочные гибкость и обслуживание приложений.</span><span class="sxs-lookup"><span data-stu-id="2d898-126">These include large and complex applications with multiple evolving subsystems; <span data-ttu-id="2d898-127">**Контейнеры для любого приложения.**</span><span class="sxs-lookup"><span data-stu-id="2d898-127">**Containers for any application.**</span></span> <span data-ttu-id="2d898-128">Контейнеры удобны для микрослужбами, но не являются взаимоисключающими для них.</span><span class="sxs-lookup"><span data-stu-id="2d898-128">Containers are convenient for microservices, but are not exclusive for them.</span></span> <span data-ttu-id="2d898-129">Контейнеры можно также с монолитные приложениями, включая устаревшие приложения на основе традиционных .NET Framework и модернизированный и через контейнеров Windows.</span><span class="sxs-lookup"><span data-stu-id="2d898-129">Containers can also be used with monolithic applications, including legacy applications based on the traditional .NET Framework and modernized through Windows Containers.</span></span> <span data-ttu-id="2d898-130">Преимущества использования Docker, такие как решить многие проблемы развертывания в рабочей среде и предоставление современные среды разработки и тестирования применяются для многих различных типов приложений.</span><span class="sxs-lookup"><span data-stu-id="2d898-130">The benefits of using Docker, such as solving many deployment-to-production issues and providing state of the art Dev and Test environments, apply to many different types of applications.</span></span>

<span data-ttu-id="2d898-131">**CLI и интегрированной среды разработки.**</span><span class="sxs-lookup"><span data-stu-id="2d898-131">**CLI versus IDE.**</span></span> <span data-ttu-id="2d898-132">С помощью средства Microsoft могут разрабатывать приложений .NET в контейнерах с помощью вашей предпочтительным подходом.</span><span class="sxs-lookup"><span data-stu-id="2d898-132">With Microsoft tools, you can develop containerized .NET applications using your preferred approach.</span></span> <span data-ttu-id="2d898-133">С помощью Docker CLI и кода Visual Studio можно разрабатывать с CLI и среду, в основе редактора.</span><span class="sxs-lookup"><span data-stu-id="2d898-133">You can develop with a CLI and an editor-based environment by using the Docker CLI and Visual Studio Code.</span></span> <span data-ttu-id="2d898-134">Или можно использовать подход, ориентированный на интегрированной среды разработки Visual Studio и ее уникальные возможности для Docker, например, скажем, для отладки приложения несколькими контейнера.</span><span class="sxs-lookup"><span data-stu-id="2d898-134">Or you can use an IDE-focused approach with Visual Studio and its unique features for Docker, such as like being able to debug multi-container applications.</span></span>

<span data-ttu-id="2d898-135">**Устойчивых облачных приложений.**</span><span class="sxs-lookup"><span data-stu-id="2d898-135">**Resilient cloud applications.**</span></span> <span data-ttu-id="2d898-136">В облачных систем и распределенных систем в общем случае всегда есть риск частичных сбоев.</span><span class="sxs-lookup"><span data-stu-id="2d898-136">In cloud-based systems and distributed systems in general, there is always the risk of partial failure.</span></span> <span data-ttu-id="2d898-137">Поскольку клиенты и службы являются отдельными процессами (контейнеров), службы могут не иметь возможность своевременно отклик на запрос клиента.</span><span class="sxs-lookup"><span data-stu-id="2d898-137">Since clients and services are separate processes (containers), a service might not be able to respond in a timely way to a client’s request.</span></span> <span data-ttu-id="2d898-138">Например возможно, служба работу из-за сбоя частичной или для обслуживания; шифрование секреты неактивные; и предотвращать секреты непреднамеренно утечка используемых конечного приложения.</span><span class="sxs-lookup"><span data-stu-id="2d898-147">Any secure solution must follow security best practices, such as encrypting secrets while in transit;
