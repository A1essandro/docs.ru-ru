# <a name="state-and-data-in-docker-applications"></a><span data-ttu-id="7a3e8-103">Состояние и данные в приложениях Docker</span><span class="sxs-lookup"><span data-stu-id="7a3e8-103">State and data in Docker applications</span></span>

<span data-ttu-id="7a3e8-104">Представьте себе контейнер как экземпляр процесса.</span><span class="sxs-lookup"><span data-stu-id="7a3e8-104">In most cases, you can think of a container as an instance of a process.</span></span> <span data-ttu-id="7a3e8-105">Процесс не находится в неизменном состоянии.</span><span class="sxs-lookup"><span data-stu-id="7a3e8-105">A process does not maintain persistent state.</span></span> <span data-ttu-id="7a3e8-106">И хотя контейнер может записывать данные в локальное хранилище, предполагать, что экземпляр всегда будет на месте — это все равно, что надеяться на постоянство одной ячейки памяти.</span><span class="sxs-lookup"><span data-stu-id="7a3e8-106">While a container can write to its local storage, assuming that an instance will be around indefinitely would be like assuming that a single location in memory will be durable.</span></span> <span data-ttu-id="7a3e8-107">Следует считать, что образы контейнеров, как и процессы, имеют несколько экземпляров или в конечном итоге будут завершены. Если они управляются с помощью оркестратора контейнеров, они могут перемещаться между узлами или виртуальными машинами.</span><span class="sxs-lookup"><span data-stu-id="7a3e8-107">Container images, like processes, should be assumed to have multiple instances or that they will eventually be killed; if they’re managed with a container orchestrator, it should be assumed that they might get moved from one node or VM to another.</span></span>

<span data-ttu-id="7a3e8-108">Docker имеет функцию под названием *оверлейная файловая система*.</span><span class="sxs-lookup"><span data-stu-id="7a3e8-108">Docker provides a feature named the *overlay file system*.</span></span> <span data-ttu-id="7a3e8-109">Она выполняет копирование при записи, при котором обновленная информация сохраняется в корневой файловой системе контейнера.</span><span class="sxs-lookup"><span data-stu-id="7a3e8-109">This implements a copy-on-write task that stores updated information to the root file system of the container.</span></span> <span data-ttu-id="7a3e8-110">Эта информация добавляется к изначальному образу, на котором базируется контейнер.</span><span class="sxs-lookup"><span data-stu-id="7a3e8-110">That information is in addition to the original image on which the container is based.</span></span> <span data-ttu-id="7a3e8-111">Если удалить контейнер из системы, эти изменения будут утеряны.</span><span class="sxs-lookup"><span data-stu-id="7a3e8-111">If the container is deleted from the system, those changes are lost.</span></span> <span data-ttu-id="7a3e8-112">Поэтому, хоть и возможно сохранить состояние контейнера в его локальном хранилище, создание системы по такой модели будет противоречить конструкции контейнера, состояние которого по умолчанию не отслеживается.</span><span class="sxs-lookup"><span data-stu-id="7a3e8-112">Therefore, while it is possible to save the state of a container within its local storage, designing a system around this would conflict with the premise of container design, which by default is stateless.</span></span>

<span data-ttu-id="7a3e8-113">Для управления постоянными данными в приложениях Docker используются следующие решения:</span><span class="sxs-lookup"><span data-stu-id="7a3e8-113">The following solutions are used to manage persistent data in Docker applications:</span></span>

-   <span data-ttu-id="7a3e8-114">[Тома данных](https://docs.docker.com/engine/tutorials/dockervolumes/), которые подключаются к узлу.</span><span class="sxs-lookup"><span data-stu-id="7a3e8-114">[Data volumes](https://docs.docker.com/engine/tutorials/dockervolumes/) that mount to the host.</span></span>

-   <span data-ttu-id="7a3e8-115">[Контейнеры томов данных](https://docs.docker.com/engine/tutorials/dockervolumes/#creating-and-mounting-a-data-volume-container), обеспечивающие общее хранилище для контейнеров с помощью внешнего контейнера.</span><span class="sxs-lookup"><span data-stu-id="7a3e8-115">[Data volume containers](https://docs.docker.com/engine/tutorials/dockervolumes/#creating-and-mounting-a-data-volume-container) that provide shared storage across containers using an external container.</span></span>

-   <span data-ttu-id="7a3e8-116">[Подключаемые модули томов](https://docs.docker.com/engine/tutorials/dockervolumes/), которые подключают тома к удаленным службам, обеспечивая долгосрочное хранение.</span><span class="sxs-lookup"><span data-stu-id="7a3e8-116">[Volume plugins](https://docs.docker.com/engine/tutorials/dockervolumes/) that mount volumes to remote services, providing long-term persistence.</span></span>

-   <span data-ttu-id="7a3e8-117">[Служба хранилища Azure](https://docs.microsoft.com/azure/storage/), предоставляющая геораспределенное хранилище для долгосрочного хранения данных контейнеров.</span><span class="sxs-lookup"><span data-stu-id="7a3e8-117">[Azure Storage](https://docs.microsoft.com/azure/storage/), which provides geo-distributable storage, providing a good long-term persistence solution for containers.</span></span>

-   <span data-ttu-id="7a3e8-118">Удаленные реляционные базы данных, например [базы данных SQL Azure](https://azure.microsoft.com/services/sql-database/) или базы данных NoSQL, такие как [Azure Cosmos DB](https://docs.microsoft.com/azure/cosmos-db/introduction), или службы кэша, такие как [Redis](https://redis.io/).</span><span class="sxs-lookup"><span data-stu-id="7a3e8-118">Remote relational databases like [Azure SQL Database](https://azure.microsoft.com/services/sql-database/) or NoSQL databases like [Azure Cosmos DB](https://docs.microsoft.com/azure/cosmos-db/introduction), or cache services like [Redis](https://redis.io/).</span></span>

<span data-ttu-id="7a3e8-119">Ниже подробно описаны эти решения.</span><span class="sxs-lookup"><span data-stu-id="7a3e8-119">The following provides more detail about these options.</span></span>

<span data-ttu-id="7a3e8-120">**Тома данных** — это каталоги из ОС узла, сопоставленные с каталогами в контейнерах.</span><span class="sxs-lookup"><span data-stu-id="7a3e8-120">**Data volumes** are directories mapped from the host OS to directories in containers.</span></span> <span data-ttu-id="7a3e8-121">Если код в контейнере имеет доступ к каталогу, на самом деле он обращается к каталогу на ОС узла.</span><span class="sxs-lookup"><span data-stu-id="7a3e8-121">When code in the container has access to the directory, that access is actually to a directory on the host OS.</span></span> <span data-ttu-id="7a3e8-122">Этот каталог не привязан к времени существования самого контейнера, и к каталогу может иметь доступ код, выполняемый непосредственно в ОС узла, или другой контейнер, сопоставляющийся с тем же каталогом узла.</span><span class="sxs-lookup"><span data-stu-id="7a3e8-122">This directory is not tied to the lifetime of the container itself, and the directory can be accessed from code running directly on the host OS or by another container that maps the same host directory to itself.</span></span> <span data-ttu-id="7a3e8-123">Поэтому тома данных хранят данные независимо от контейнера.</span><span class="sxs-lookup"><span data-stu-id="7a3e8-123">Thus, data volumes are designed to persist data independently of the life of the container.</span></span> <span data-ttu-id="7a3e8-124">Если удалить контейнер или образ из узла Docker, данные из томов данных не будут удалены.</span><span class="sxs-lookup"><span data-stu-id="7a3e8-124">If you delete a container or an image from the Docker host, the data persisted in the data volume is not deleted.</span></span> <span data-ttu-id="7a3e8-125">Доступ к данным в томе можно также получить из ОС узла.</span><span class="sxs-lookup"><span data-stu-id="7a3e8-125">The data in a volume can be accessed from the host OS as well.</span></span>

<span data-ttu-id="7a3e8-126">**Контейнеры томов данных** — это усовершенствованная версия обычных томов данных.</span><span class="sxs-lookup"><span data-stu-id="7a3e8-126">**Data volume containers** are an evolution of regular data volumes.</span></span> <span data-ttu-id="7a3e8-127">Контейнер томов данных — это простой контейнер, содержащий один или несколько томов данных.</span><span class="sxs-lookup"><span data-stu-id="7a3e8-127">A data volume container is a simple container that has one or more data volumes within it.</span></span> <span data-ttu-id="7a3e8-128">Контейнер томов данных предоставляет контейнерам доступ из центральной точки подключения.</span><span class="sxs-lookup"><span data-stu-id="7a3e8-128">The data volume container provides access to containers from a central mount point.</span></span> <span data-ttu-id="7a3e8-129">Это удобный метод доступа к данным, поскольку не учитывается расположение исходных данных.</span><span class="sxs-lookup"><span data-stu-id="7a3e8-129">This method of data access is convenient because it abstracts the location of the original data.</span></span> <span data-ttu-id="7a3e8-130">В остальном он почти не отличается от обычного тома данных, и данные хранятся в отдельном контейнере независимо от жизненного цикла контейнеров приложений.</span><span class="sxs-lookup"><span data-stu-id="7a3e8-130">Other than that, its behavior is similar to that of a regular data volume, so data is persisted in this dedicated container independently of the lifecycle of the application’s containers.</span></span>

<span data-ttu-id="7a3e8-131">Как показано на рисунке 4-5, обычные тома Docker могут храниться за пределами самих контейнерами, но в физических границах сервера узла или виртуальной машины.</span><span class="sxs-lookup"><span data-stu-id="7a3e8-131">As shown in Figure 4-5, regular Docker volumes can be stored outside of the containers themselves but within the physical boundaries of the host server or VM.</span></span> <span data-ttu-id="7a3e8-132">Тем не менее контейнеры Docker с одного сервера узла или виртуальной машины не могут обращаться к тому на другом сервере узла или виртуальной машине.</span><span class="sxs-lookup"><span data-stu-id="7a3e8-132">However, Docker containers cannot access a volume from one host server or VM to another.</span></span> <span data-ttu-id="7a3e8-133">Другими словами, при использовании этих томов невозможно управлять данными контейнеров, расположенных на разных узлах Docker</span><span class="sxs-lookup"><span data-stu-id="7a3e8-133">In other words, with these volumes, it is not possible to manage data shared between containers that run on different Docker hosts</span></span>

<span data-ttu-id="7a3e8-136">Кроме того, когда контейнеры Docker управляются оркестратором, контейнеры могут "перемещаться" между узлами в рамках оптимизации, выполняемой кластером.</span><span class="sxs-lookup"><span data-stu-id="7a3e8-136">In addition, when Docker containers are managed by an orchestrator, containers might “move” between hosts, depending on the optimizations performed by the cluster.</span></span> <span data-ttu-id="7a3e8-137">Поэтому не рекомендуется использовать тома данных для бизнес-данных.</span><span class="sxs-lookup"><span data-stu-id="7a3e8-137">Therefore, it is not recommended that you use data volumes for business data.</span></span> <span data-ttu-id="7a3e8-138">Но это хороший инструмент для работы с файлами трассировки, временными файлами или подобными элементами, которые не влияют на целостность бизнес-данных.</span><span class="sxs-lookup"><span data-stu-id="7a3e8-138">But they are a good mechanism to work with trace files, temporal files, or similar that will not impact business data consistency.</span></span>

<span data-ttu-id="7a3e8-139">**Подключаемые модули томов**, например [Flocker](https://clusterhq.com/flocker/), предоставляют доступ к данным на всех узлах в кластере.</span><span class="sxs-lookup"><span data-stu-id="7a3e8-139">**Volume plugins** like [Flocker](https://clusterhq.com/flocker/) provide data access across all hosts in a cluster.</span></span> <span data-ttu-id="7a3e8-140">Не все подключаемые модули томов создаются одинаково, но обычно являются надежным внешним хранилищем данных для неизменных контейнеров.</span><span class="sxs-lookup"><span data-stu-id="7a3e8-140">While not all volume plugins are created equally, volume plugins typically provide externalized persistent reliable storage from immutable containers.</span></span>

<span data-ttu-id="7a3e8-141">**Удаленные источники данных и инструменты кэширования**, такие как база данных Azure SQL, Azure Cosmos DB или служба кэша, например Redis, можно использовать в упакованных в контейнеры приложениях так же, как они используются при разработке без контейнеров.</span><span class="sxs-lookup"><span data-stu-id="7a3e8-141">**Remote data sources and cache** tools like Azure SQL Database, Azure Cosmos DB, or a remote cache like Redis can be used in containerized applications the same way they are used when developing without containers.</span></span> <span data-ttu-id="7a3e8-142">Это проверенный способ хранения данных бизнес-приложений.</span><span class="sxs-lookup"><span data-stu-id="7a3e8-142">This is a proven way to store business application data.</span></span>

<span data-ttu-id="7a3e8-143">**Служба хранилища Azure**</span><span class="sxs-lookup"><span data-stu-id="7a3e8-143">**Azure Storage.**</span></span> <span data-ttu-id="7a3e8-144">Бизнес-данные, как правило, необходимо хранить во внешних ресурсах или базах данных, например в службе хранилища Azure.</span><span class="sxs-lookup"><span data-stu-id="7a3e8-144">Business data usually will need to be placed in external resources or databases, like Azure Storage.</span></span> <span data-ttu-id="7a3e8-145">Служба хранилища Azure предоставляет следующие службы в облаке:</span><span class="sxs-lookup"><span data-stu-id="7a3e8-145">Azure Storage, in concrete, provides the following services in the cloud:</span></span>

-   <span data-ttu-id="7a3e8-146">Хранилище BLOB-объектов хранит неструктурированные данные объектов.</span><span class="sxs-lookup"><span data-stu-id="7a3e8-146">Blob storage stores unstructured object data.</span></span> <span data-ttu-id="7a3e8-147">Большой двоичный объект может представлять собой текстовые или двоичные данные любого типа, например документ или файл мультимедиа (файлы с изображением, аудио и видео).</span><span class="sxs-lookup"><span data-stu-id="7a3e8-147">A blob can be any type of text or binary data, such as document or media files (images, audio, and video files).</span></span> <span data-ttu-id="7a3e8-148">Хранилище BLOB-объектов также называют хранилищем объектов.</span><span class="sxs-lookup"><span data-stu-id="7a3e8-148">Blob storage is also referred to as Object storage.</span></span>

-   <span data-ttu-id="7a3e8-149">Хранилище файлов обеспечивает хранение приложений прежних версий по стандартному протоколу SMB.</span><span class="sxs-lookup"><span data-stu-id="7a3e8-149">File storage offers shared storage for legacy applications using standard SMB protocol.</span></span> <span data-ttu-id="7a3e8-150">Виртуальные машины Azure и облачные службы могут совместно использовать данные файлов в компонентах приложений через подключенный общий ресурс.</span><span class="sxs-lookup"><span data-stu-id="7a3e8-150">Azure virtual machines and cloud services can share file data across application components via mounted shares.</span></span> <span data-ttu-id="7a3e8-151">Локальные приложения могут обращаться к данным файлов в общем ресурсе через файловую службу REST API.</span><span class="sxs-lookup"><span data-stu-id="7a3e8-151">On-premises applications can access file data in a share via the File service REST API.</span></span>
