# <a name="microservices-addressability-and-the-service-registry"></a>Возможность адресации Микрослужбами и реестр службы

Каждый микрослужбу имеет уникальное имя (URL), который используется для разрешения расположения. Ваш микрослужбу должен быть адресуемый везде, где она запущена. Если нужно думать о какой компьютер работает под управлением определенной микрослужбу, могут возникнуть неправильный быстро. Таким же образом, служба DNS сопоставляет URL-адреса к определенному компьютеру ваш микрослужбу должен иметь уникальное имя, чтобы функция обнаружения текущего места. Микрослужбами должны адресуемых имен, которые их независимо от инфраструктуры, в которой они выполняются на. Это значит, что взаимодействие между развертывание службы и как ее обнаружения, так как должен быть [реестр службы](http://microservices.io/patterns/service-registry.html). В том же ключе при сбое компьютера, служба реестра должна быть возможность указывать, где выполняется служба.

[Шаблон службы реестра](http://microservices.io/patterns/service-registry.html) является ключевой частью службы обнаружения. Реестр является база данных, содержащая сетевые расположения экземпляров службы. Реестр службы должен быть высокодоступной и в актуальном состоянии. Клиенты могут кэшировать сетевые расположения, полученная из реестра службы. Однако эти сведения со временем устаревает, и клиентов не сможет обнаруживать экземпляров службы. Таким образом реестр службы состоит из кластера серверов, которые используют протокол репликации для поддержания согласованности.

Обнаружение службы встроена в некоторых средах развертывания микрослужбу в (называемые кластера, чтобы рассматриваться в следующем разделе). Например в среде Azure контейнера службы Kubernetes DC/OS с Marathon процессом службы регистрации экземпляра и отмену регистрации. Они также выполняются прокси-сервера на каждом узле кластера, который играет роль маршрутизатора обнаружение на стороне сервера. Другой пример — Azure Service Fabric также предоставляет реестр службы через ее службы имен out of box.

Обратите внимание, что некоторые сходства, между реестр службы и шаблон шлюза API, который помогает решить эту проблему также. Например [прокси-сервера службы структуры обратного](https://docs.microsoft.com/azure/service-fabric/service-fabric-reverseproxy) — это тип реализации API шлюз, основанный на службы Fabrice именования и помогает устранить разрешения адреса для внутренних служб.
