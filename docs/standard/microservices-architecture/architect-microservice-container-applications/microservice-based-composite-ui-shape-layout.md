# <a name="creating-composite-ui-based-on-microservices-including-visual-ui-shape-and-layout-generated-by-multiple-microservices"></a><span data-ttu-id="0a688-104">Создание составного пользовательского интерфейса на основе микрослужб, включая создание его визуальной формы и структуры с помощью нескольких микрослужб</span><span class="sxs-lookup"><span data-stu-id="0a688-104">Creating composite UI based on microservices, including visual UI shape and layout generated by multiple microservices</span></span>

<span data-ttu-id="0a688-105">Архитектура микрослужб часто начинается с того, что сервер обрабатывает данные и логику.</span><span class="sxs-lookup"><span data-stu-id="0a688-105">Microservices architecture often starts with the server side handling data and logic.</span></span> <span data-ttu-id="0a688-106">Однако более сложный подход — разработать пользовательский интерфейс приложения также на основе микрослужб.</span><span class="sxs-lookup"><span data-stu-id="0a688-106">However, a more advanced approach is to design your application UI based on microservices as well.</span></span> <span data-ttu-id="0a688-107">Это означает, что у вас появляется составной пользовательский интерфейс, создаваемый микрослужбами, вместо размещения микрослужб на сервере и наличия монолитного клиентского приложения, которое использует эти микрослужбы.</span><span class="sxs-lookup"><span data-stu-id="0a688-107">That means having a composite UI produced by the microservices, instead of having microservices on the server and just a monolithic client app consuming the microservices.</span></span> <span data-ttu-id="0a688-108">При использовании этого подхода создаваемые микрослужбы можно дополнить логикой и визуальным представлением.</span><span class="sxs-lookup"><span data-stu-id="0a688-108">With this approach, the microservices you build can be complete with both logic and visual representation.</span></span>

<span data-ttu-id="0a688-109">На рис. 4-20 показан более простой подход, при котором монолитное клиентское приложение просто использует микрослужбы.</span><span class="sxs-lookup"><span data-stu-id="0a688-109">Figure 4-20 shows the simpler approach of just consuming microservices from a monolithic client application.</span></span> <span data-ttu-id="0a688-110">Конечно, вы можете использовать промежуточную службу ASP.NET MVC между формированием HTML и JavaScript.</span><span class="sxs-lookup"><span data-stu-id="0a688-110">Of course, you could have an ASP.NET MVC service in between producing the HTML and JavaScript.</span></span> <span data-ttu-id="0a688-111">На рисунке показана упрощенная схема, которая выделяет наличие единого (монолитного) пользовательского интерфейса клиента, использующего микрослужбы. Эта схема делает акцент на логике и данных, а не на форме пользовательского интерфейса (HTML и JavaScript).</span><span class="sxs-lookup"><span data-stu-id="0a688-111">The figure is a simplification that highlights that you have a single (monolithic) client UI consuming the microservices, which just focus on logic and data and not on the UI shape (HTML and JavaScript).</span></span>

<span data-ttu-id="0a688-112">**Рис. 4-20**.</span><span class="sxs-lookup"><span data-stu-id="0a688-112">**Figure 4-20**.</span></span> <span data-ttu-id="0a688-113">Приложение с монолитным пользовательским интерфейсом, использующее серверные микрослужбы</span><span class="sxs-lookup"><span data-stu-id="0a688-113">A monolithic UI application consuming back-end microservices</span></span>

<span data-ttu-id="0a688-114">В отличие от монолитного пользовательского интерфейса составной пользовательский интерфейс создается и формируется самими микрослужбами.</span><span class="sxs-lookup"><span data-stu-id="0a688-114">In contrast, a composite UI is precisely generated and composed by the microservices themselves.</span></span> <span data-ttu-id="0a688-115">Некоторые микрослужбы отвечают за внешний вид определенных областей пользовательского интерфейса.</span><span class="sxs-lookup"><span data-stu-id="0a688-115">Some of the microservices drive the visual shape of specific areas of the UI.</span></span> <span data-ttu-id="0a688-116">Основное различие состоит в том, что у вас есть компоненты пользовательского интерфейса клиента на основе шаблонов (например, классы TS), и модель представления, определяющая форму данных пользовательского интерфейса для этих шаблонов, поступает от каждой микрослужбы.</span><span class="sxs-lookup"><span data-stu-id="0a688-116">The key difference is that you have client UI components (TS classes, for example) based on templates, and the data-shaping-UI ViewModel for those templates comes from each microservice.</span></span>

<span data-ttu-id="0a688-117">Во время запуска приложения клиента каждый из компонентов пользовательского интерфейса клиента (например, классы TypeScript) регистрируется в микрослужбе инфраструктуры, которая предоставляет модели представления для данного сценария.</span><span class="sxs-lookup"><span data-stu-id="0a688-117">At client application start-up time, each of the client UI components (TypeScript classes, for example) registers itself with an infrastructure microservice capable of providing ViewModels for a given scenario.</span></span> <span data-ttu-id="0a688-118">Если микрослужба изменяет форму интерфейса, это приводит к изменению пользовательского интерфейса.</span><span class="sxs-lookup"><span data-stu-id="0a688-118">If the microservice changes the shape, the UI changes also.</span></span>

<span data-ttu-id="0a688-119">На рисунке 4-21 показан вариант составного пользовательского интерфейса.</span><span class="sxs-lookup"><span data-stu-id="0a688-119">Figure 4-21 shows a version of this composite UI approach.</span></span> <span data-ttu-id="0a688-120">Это упрощенный вариант, так как у вас могут использоваться другие микрослужбы, агрегирующие отдельные фрагменты на основе различных методов, — это зависит от того, используете ли вы традиционный подход к созданию веб-приложения (ASP.NET MVC) или одностраничное приложение (SPA).</span><span class="sxs-lookup"><span data-stu-id="0a688-120">This is simplified, because you might have other microservices that are aggregating granular parts based on different techniques—it depends on whether you are building a traditional web approach (ASP.NET MVC) or an SPA (Single Page Application).</span></span>

![](./media/image21.png)

<span data-ttu-id="0a688-121">**Рис. 4-21**.</span><span class="sxs-lookup"><span data-stu-id="0a688-121">**Figure 4-21**.</span></span> <span data-ttu-id="0a688-122">Пример составного пользовательского интерфейса приложения, форма которого определяется серверными микрослужбами</span><span class="sxs-lookup"><span data-stu-id="0a688-122">Example of a composite UI application shaped by back-end microservices</span></span>

<span data-ttu-id="0a688-123">Каждая из этих микрослужб, составляющих пользовательский интерфейс, похожа на небольшой шлюз API.</span><span class="sxs-lookup"><span data-stu-id="0a688-123">Each of those UI composition microservices would be similar to a small API Gateway.</span></span> <span data-ttu-id="0a688-124">Однако в этом случае каждая микрослужба отвечает за небольшую область пользовательского интерфейса.</span><span class="sxs-lookup"><span data-stu-id="0a688-124">But in this case each is responsible for a small UI area.</span></span>

<span data-ttu-id="0a688-125">Подход с составным пользовательским интерфейсом, который формируется микрослужбами, может быть более или менее трудным в зависимости от того, какие методы создания пользовательского интерфейса применяются.</span><span class="sxs-lookup"><span data-stu-id="0a688-125">A composite UI approach that is driven by microservices can be more challenging or less so, depending on what UI technologies you are using.</span></span> <span data-ttu-id="0a688-126">Например, вы не сможете использовать одни и те же методы для создания традиционного веб-приложения, одностраничного приложения или собственного мобильного приложения (как при разработке приложений Xamarin, для которых этот подход может оказаться более трудным).</span><span class="sxs-lookup"><span data-stu-id="0a688-126">For instance, you will not use the same techniques for building a traditional web application that you use for building an SPA or for native mobile app (as when developing Xamarin apps, which can be more challenging for this approach).</span></span>

<span data-ttu-id="0a688-127">В примере приложения [EShopOnContainers](http://aka.ms/MicroservicesArchitecture) используется монолитный пользовательский интерфейс по различным причинам.</span><span class="sxs-lookup"><span data-stu-id="0a688-127">The [eShopOnContainers](http://aka.ms/MicroservicesArchitecture) sample application uses the monolithic UI approach for multiple reasons.</span></span> <span data-ttu-id="0a688-128">Во-первых, он позволяет новичкам познакомиться с микрослужбами и контейнерами.</span><span class="sxs-lookup"><span data-stu-id="0a688-128">First, it is an introduction to microservices and containers.</span></span> <span data-ttu-id="0a688-129">Подход с составным пользовательским интерфейсом более сложен, но он еще больше усложняется при проектировании и разработки пользовательского интерфейса.</span><span class="sxs-lookup"><span data-stu-id="0a688-129">A composite UI is more advanced but also requires further complexity when designing and developing the UI.</span></span> <span data-ttu-id="0a688-130">Во-вторых, eShopOnContainers также предоставляет собственное мобильное приложение на основе Xamarin, что усложнило бы разработку на C\# для клиента.</span><span class="sxs-lookup"><span data-stu-id="0a688-130">Second, eShopOnContainers also provides a native mobile app based on Xamarin, which would make it more complex on the client C\# side.</span></span>
