# <a name="creating-composite-ui-based-on-microservices-including-visual-ui-shape-and-layout-generated-by-multiple-microservices"></a>Создание составного пользовательского интерфейса на основании микрослужбами, включая visual пользовательского интерфейса формы и структуры, созданные несколько микрослужбами

Архитектура Микрослужбами часто начинается со стороны сервера, обработки данных и логику. Тем не менее более сложные подход заключается в том, в разработке пользовательского интерфейса на основании также микрослужбами приложения. Что означает, что составной пользовательского интерфейса, созданные микрослужбами, вместо того, микрослужбами на сервере и просто использование микрослужбами монолитные клиентского приложения. При таком подходе микрослужбами, построении можно с логикой и визуальное представление.

Рис. 4-20 показывает более простой подход с только что использование микрослужбами из монолитные клиентского приложения. Конечно может использовать службу ASP.NET MVC между формирование HTML и JavaScript. На рисунке показан упрощение, которая выделяет наличие одного клиента (монолитные) пользовательского интерфейса, использование микрослужбами, который Обратите особое внимание на логику и данные, а не на форму пользовательского интерфейса (HTML и JavaScript).

![](./media/image20.png)

**Рис. 4-20**. Монолитные пользовательского интерфейса приложения, использующего серверную микрослужбами

В отличие от этого составного пользовательского интерфейса точно создается и сформирован микрослужбами, сами. Некоторые микрослужбами диска visual форму определенных областях пользовательского интерфейса. Ключевое различие состоит в наличии клиентских компонентов пользовательского интерфейса (например, классы служб Терминалов) на основе шаблонов, что ViewModel формирования интерфейса данных для этих шаблонов поступают из каждого микрослужбу.

Во время запуска приложения клиента каждый из компонентов пользовательского интерфейса клиента (например, классы TypeScript) регистрируется с микрослужбу инфраструктуры, подающего ViewModels для данного сценария. Если микрослужбу приводит к изменению формы, также изменения пользовательского интерфейса.

На рисунке 4-21 показана версия этого составного подхода пользовательского интерфейса. Это существенно упрощена, поскольку может иметь другие микрослужбами, статистической обработке детализированные части на основании различных методов — это зависит от создаваемого подход с использованием традиционных web (ASP.NET MVC) или SPA (одностраничного приложения).

![](./media/image21.png)

**На рисунке 4-21**. Пример составного пользовательского интерфейса приложения были сформированы по микрослужбами серверной части

Каждый из этих микрослужбами композиции пользовательского интерфейса будет совпадать с небольшой шлюза API. Однако в этом случае каждый отвечает за небольшой области пользовательского интерфейса.

Составной подход пользовательского интерфейса, который управляется микрослужбами может быть более сложной задачей или менее таким образом, вы используете в зависимости от того, какие технологии пользовательского интерфейса. Для экземпляра, вы не будете использовать те же методы для создания традиционных веб-приложения, используемого для формирования SPA или для собственного мобильного приложения (как при разработке приложения Xamarin, которые могут быть более сложной задачей для данного подхода).

[EShopOnContainers](http://aka.ms/MicroservicesArchitecture) примере приложение использует объединенный подход пользовательского интерфейса по нескольким причинам. Во-первых это введение микрослужбами и контейнеры. Составной пользовательский Интерфейс более сложных, но также требует дальнейшего сложности при проектировании и разработке пользовательского интерфейса. Во-вторых, eShopOnContainers также предоставляет собственный мобильные приложения с учетом Xamarin, который позволил бы более сложные на клиенте C\# стороне.

Тем не менее мы рекомендуем вам использовать следующие ссылки на дополнительные сведения о составных пользовательского интерфейса на основании микрослужбами.
