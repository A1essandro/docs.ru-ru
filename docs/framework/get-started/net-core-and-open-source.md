# Ядро .NET и открытый исходный код
.NET Core — это модульная версия .NET Framework, которую можно переносить на другие платформы для увеличения повторного и совместного использования кода. Кроме того, .NET Core будет поддерживать открытый исходный код и принимать работы участников сообщества. В этом разделе даются ответы на общие вопросы о .NET Core и предоставляются сведения о доступе к пакетам с открытым исходным кодом и их отправке.  
  
<a name="BKMK_WhatisNETCore"></a>   
## Что такое .NET Core?  
 Как было упомянуто выше, .NET Core — это модульная версия .NET Framework, которую можно переносить на другие платформы.  
  
 .NET Core можно переносить на другие платформы, так как несмотря на то, что она является подмножеством полной версии .NET Framework, она предоставляет ключевые возможности для реализации функций приложений, необходимых для повторного использования этого кода независимо от целевой платформы. Раньше в разных версиях .NET для разных платформ отсутствовали общие функции для ключевых задач, таких как чтение локальных файлов. Доступные целевые платформы Майкрософт для .NET Core включают традиционную настольную систему Windows, а также устройства и телефоны под управлением ОС Windows. При использовании сторонних средств, таких как Xamarin, требуется возможность переноса .NET Core на устройства IOS и Android. Кроме того, среда .NET Core будет скоро доступна для операционных систем Mac и Linux, позволяющих запускать веб\-приложения на этих системах.  
  
 .NET Core является модульной средой, так как она выпускается в NuGet в небольших пакетах сборки. .NET Core предоставляется в виде небольших пакетов с компонентами, а не в одной большой сборке, которая содержит большинство основных функций. Это позволяет использовать более гибкую модель разработки и выбирать необходимые функции для приложений и библиотек. Дополнительные сведения о пакетах .NET, выпускаемых в NuGet, см. в разделе [.NET Framework и внештатные выпуски](../../../docs/framework/get-started/the-net-framework-and-out-of-band-releases.md).  
  
 Ниже представлены некоторые часто задаваемые вопросы о .NET Core.  
  
### Как лучше всего использовать .NET Core?  
 Для существующих приложений использование переносимых библиотек классов, проектов универсальных приложений и отдельной от специального кода платформы бизнес\-логики — лучший способ получить преимущества .NET Core и увеличить повторное использование кода. Для приложений можно выбрать шаблоны Model\-View\-Controller \(MVC\) или Model View\-ViewModel \(MVVM\) для простой миграции приложений на .NET Core.  
  
### Как .NET Core влияет на совместимость и развертывание?  
 Для развертывания существует несколько разных сценариев, но оно не должно вызывать сложностей. Платформа .NET Framework будет распределена в Windows и обновлена с помощью Центра обновления Майкрософт. В некоторых случаях приложение будет зависеть от развертывания на диске .NET Framework, а в других случаях — от сборок .NET, которые развертываются в пакете приложений. Кроме того, совместимость версий все еще приоритетна для .NET Framework, поэтому, если приложение выполняется на более новой версии сборки по сравнению с версией при компиляции, его поведение не должно измениться.  
  
|Сценарий|Развертывание|  
|--------------|-------------------|  
|Классические приложения Windows и приложения ASP.NET, которые выполняются на компьютерах Windows|Развертывание выполняется так же. Приложение зависит от установки  .NET Framework на сервере или на компьютере пользователя. Если платформа .NET Framework не установлена, пользователю будет предложено установить ее. Кроме того, вы можете объединить в цепочку установку .NET с приложением. При использовании сборок из .NET Core, которые не включены в .NET Framework, эти сборки включаются в пакет приложения. Также если приложение ссылается на две версии одной сборки, приложение будет перенаправлено на более новую сборку. Дополнительные сведения см. в разделах [Перенаправление версий сборки на уровне приложения](../../../docs/framework/configure-apps/redirect-assembly-versions.md#BKMK_Redirectingassemblyversionsattheapplevel) и [Практическое руководство. |Приложения Windows и Windows Phone, доступные в Магазинах Windows и Windows Phone|Процесс развертывания не изменился, и приложения используют сборки .NET Framework, включенные в операционную систему. Если приложение использует функции, выпущенные в сборке .NET Core, которая не входит в .NET Framework, сборка будет включена в пакет приложений. Если приложение ссылается на несколько версий одной сборки, это приложение автоматически будет использовать более новую версию сборки.|

Это может измениться, так как все больше сборок .NET Core включают открытый исходный код.|  
  
<a name="BKMK_NETCoreOpenSourcePackages"></a>   
## Пакеты с открытым исходным кодом .NET Core  
 В дополнение к использованию модулей в .NET Framework Майкрософт публикует пакеты .NET Core с открытым исходным кодом на сайте [GitHub](https://github.com/) под лицензией [MIT](https://github.com/dotnet/corefx/blob/master/LICENSE). Это означает, что вы можете клонировать репозиторий Git, читать и компилировать код и отправлять запросы на включение внесенных изменений, как в других пакетах с открытым исходным кодом, которые можно найти в GitHub. Так как мы стремимся обеспечить качество и совместимость, каждый запрос на включение внесенных изменений будет тщательно оценен, перед тем как он будет принят. Ниже представлены некоторые часто задаваемые вопросы.  
  
### Как получить код и выполнить его сборку?  
 Исходные пакеты .NET Core хранятся в `GitHub`, где `Git` используется в качестве системы управления версиями. Чтобы получить доступ к коду и выполнить его сборку, следует ознакомиться с основами [Git](http://git-scm.com/), [GitHub](https://github.com/dotnet/corefx) и [Visual Studio](http://msdn.microsoft.com/vstudio/aa718325.aspx), однако в следующих шагах представлены некоторые сведения и ссылки для начала работы.  
  
 Сведения о настройке Git и GitHub см. в статье [Настройка Git](https://help.github.com/articles/set-up-git/) на сайте GitHub.  
  
 Чтобы получить доступ к коду .NET Framework, требуется разветвление репозитория Git, который содержит этот код и клонирует его на компьютер разработки. Чтобы выполнить разветвление кода, перейдите в репозиторий и щелкните **Разветвление** в правом верхнем углу браузера. Вы можете клонировать разветвление, используя приложение GitHub или командную строку в оболочке GitHub. Чтобы клонировать репозиторий в оболочке GitHub, выполните следующую команду:  
  
```  
git clone https://github.com/dotnet/corefx  
```  
  
 Чтобы создать код, требуется установленная версия Visual Studio 2013. Вы можете создать решение в Visual Studio 2013 и нажать клавишу F5 или создать его в командной строке разработчика. Для создания с помощью командной строки откройте командную строку разработчика и перейдите в папку, в которой клонировали исходный код. Затем введите:  
  
```  
build.cmd  
  
```  
  
 Дополнительные сведения о том, как получить доступ к командной строке разработчика, см. в разделе [Командные строки](../../../docs/framework/tools/developer-command-prompt-for-vs.md).  
  
### Как рекомендуется отправлять код?  
 Перед тем как мы сможем внедрить ваш код в главную ветвь, вам необходимо подписать [Лицензионное соглашение участника \(CLA\)](https://cla.dotnetfoundation.org/).  
  
 Мы принимаем работы от сообщества, в которых используется разветвление и модель запроса на включение внесенных изменений. Вам следует выполнить разветвление и клонировать код, а затем отправить запрос на включение внесенных изменений в главную ветвь .NET Framework. Дополнительные сведения о запросах на включение внесенных изменений см. в разделе [Использование запросов на включение внесенных изменений](https://help.github.com/articles/using-pull-requests/). Подробные рекомендации по отправке запросов на включение внесенных изменений см. в разделе [Руководство по участию в .NET Core](https://github.com/dotnet/corefx/wiki/Contributing).  
  
### Почему вы не приняли мой запрос на включение внесенных изменений?  
 Если мы отклоняем запрос на включение внесенных изменений, мы указываем причину, но в целом следует знать, что Майкрософт следит за качеством и совместимостью кода. Если запрос на включение внесенных изменений не был принят, возможно, вы не придерживались рекомендаций по кодированию, не провели тестирование перед отправкой кода или нарушили совместимость API. Кроме того, изменения кода должны соответствовать схеме .NET Framework. Рекомендации по созданию кода и дополнительные сведения см. в статье [Руководство по участию в .NET Core](https://github.com/dotnet/corefx/wiki/Contributing).
