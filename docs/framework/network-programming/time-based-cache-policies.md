# Политики кэша на основе времени
Time\- на основе политики кэша, указывающее свежесть кэшированных записей с помощью времени ресурс был получать заголовки, возвращенные с ресурсом и текущим временем.  При установлении Time\- на основе политики кэша, можно либо использовать политику Time\-, основанную <xref:System.Net.Cache.HttpRequestCacheLevel> или создать настраиванную Time\- на основе политики.  При использовании по умолчанию на основе политик Time\- ресурсов, получаемых с помощью протокола HTTP \(HTTP\), точная функциональности кэша определяется в заголовках включенными в кэшированном ответе и расширениях функциональности, определенными в шагах 13 и 14 RFC 2616, [http:\/\/www.ietf.org](http://www.ietf.org/) доступном по адресу.  Пример кода, демонстрирующий установки по умолчанию Time\- на политику для ресурсов HTTP см. в разделе [Практическое руководство. Установите Time\- Based значение по умолчанию политику кэша для приложения](../../../docs/framework/network-programming/how-to-set-the-default-time-based-cache-policy-for-an-application.md).  Примеры кода, демонстрирующие создание и использование политики кэша см. в разделе [Настройка кэширования в приложениях сети](../../../docs/framework/network-programming/configuring-caching-in-network-applications.md).  
  
## Критерии для определения свежесть кэшированных записей  
 Настраивать Time\- на основе политики кэша, можно задать одно или несколько из следующих условий, используемых для указания свежесть кэшированных записей.  
  
-   Максимальную длительность  
  
-   Максимальная черствость  
  
-   Минимальная свежесть  
  
-   Дата синхронизации кэша  
  
> [!NOTE]
>  Использование Time\- based значение по умолчанию политика кэша путать с параметром по умолчанию политика кэша для приложения.  По умолчанию Time\- на основе политики определенной политики, которые могут быть использованы по запросу или уровня приложения.  По умолчанию политика кэша для приложения Time\- место хранения\-, основанной на основе политики \(или\), вступает в силу при отсутствии политика не будет установлена на запрос.  Дополнительные сведения об установке по умолчанию политика кэша для приложения см. в разделе <xref:System.Net.WebRequest.DefaultCachePolicy%2A>.  
  
### Максимальный возраст  
 Максимальная критериям политики длительности определяет количество времени кэшированную копию ресурса может быть использован.  Если доступна кэшированная копия ресурса дольше заданного периода времени, ресурс должен быть revalidated, проверяя его в отношении содержимого на сервере.  Если максимальная длительность разрешитьTfа бы ресурс, который нужно использовать, после этого условия не истекает, удостоино если максимальное значение черствости также не указано.  
  
### Максимальная черствость  
 Максимальная условий политики черствости указывающее продолжительность времени после содержимого, срок действия кэшированной копии ресурсов можно использовать.  Это единственная критериям политики кэша, которая позволяет использовать ресурсы после того, как они истек.  
  
### Минимальная свежесть  
 Минимальная критериям политики актуальности задает продолжительность времени до истечения срока действия содержимого, кэшированную копию ресурса может быть использован.  Эта политика влияет вызывать запись кэша до даты окончания его срока действия. поэтому минимальные и максимальные возможности свежесть параметры черствости являются взаимоисключающими.  
  
## Дата синхронизации кэша  
 Критерии политики даты синхронизации кэша указывает, когда кэшированная копия ресурса должна быть revalidated путем проверки ее в отношении содержимого на сервере.  Если содержимое изменилось, поскольку элемент был кэширован, он извлечен из сервера, сохраняется в кэше, и возвращается к приложению.  Если содержимое не изменилось, то его отметка времени обновлена и приложение получает кэшированное содержимое.  
  
 Дата синхронизации кэша позволяет указать абсолютную дату сверки кэшированного содержимого.  Если свежая запись кэша в последний раз была revalidated до даты синхронизации кэша, повторной проверки с сервером по\-прежнему происходит.  Если запись была revalidated после того, как дата синхронизации кэша и нет дополнительных требований к повторной проверки актуальности или сервера, что сделает кэшированную запись, запись из кэша.  Если дата синхронизации кэша устанавливается на будущую дату, сверка записи кэша выполняется каждый раз при запросе, пока не пройдет дата синхронизации кэша.
